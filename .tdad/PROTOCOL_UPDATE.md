# آلية تحديث هيكل TDAD عند التطوير

## الهدف
تحديث ملفات `.workflow.json` أولاً بأول عند إضافة أو تعديل سلوك في النظام، ليبقى الرسم التبعيات (Dependency Graph) مطابقاً للكود.

---

## مرجع التقنيات والكود (مصدر الحقيقة)

**قبل إضافة أو وصف أي endpoint أو صفحة:** راجع [TECH_STACK_AND_CODE_REFERENCE.md](./TECH_STACK_AND_CODE_REFERENCE.md).

هذا الملف مستخرج من الكود الفعلي ويحتوي على: التقنيات الفعلية (إصدارات Angular، Hono، Drizzle، PostgreSQL، JWT)، مسارات الـ API (البادئة `/api`، Auth، Dashboard، مسارات الأعمال)، المحركات (transaction، workflow، reporting، inventory، currency، ui-builder، websocket)، الجداول، مسارات الواجهة والحراس، والـ middleware.

عند وصف endpoint في عقدة، استخدم **نفس المسار والطريقة** كما في المرجع (مثلاً `GET /api/businesses/:bizId/salaries`).

---

## القاعدة الذهبية
**قبل أو فور إنهاء أي سلوك قابل للاختبار (feature):**
1. حدد هل السلوك **API** أم **UI** أم **كلاهما**.
2. حدد المجلد المناسب في الهيكل (backend/xxx أو frontend/xxx).
3. أضف أو حدّث **عقدة واحدة** في الملف المناسب، ثم احفظ.

---

## خطوات التحديث حسب نوع التطوير

### 1) عند إضافة endpoint جديد في Backend

| الخطوة | الإجراء |
|--------|---------|
| 1 | افتح `.tdad/workflows/backend/backend.workflow.json` — إن لم يكن للمجال مجلد، أضف عقدة `nodeType: "folder"` مع `folderPath: "backend/اسم-المجال"`. |
| 2 | إن كان المجلد موجوداً، افتح `.tdad/workflows/backend/اسم-المجال/اسم-المجال.workflow.json`. إن لم يكن الملف موجوداً، أنشئه بنفس صيغة الملفات الموجودة. |
| 3 | أضف عقدة جديدة `nodeType: "feature"`: |
| | • `id`: فعل-اسم بصيغة kebab (مثل `get-orders-list`, `post-voucher-status-cancelled`) |
| | • `title`: اسم واضح للميزة |
| | • `description`: وصف BDD (متى، ماذا، نجاح، فشل) — انظر القالب أدناه |
| | • `fileName`: نفس `id` |
| | • `position`: { x, y } مختلف عن العقد الأخرى |
| | • `dependencies`: مصفوفة IDs للعقد التي تعتمد عليها هذه الميزة (إن وجدت) |
| | • `testLayers`: `["api"]` |
| 4 | إن وُجد تبعية داخل نفس المجلد، أضف حافة في `edges`: `{ "id": "من-إلى", "source": "id-العقدة-المصدر", "target": "id-العقدة-الهدف" }`. |

**قالب وصف API:**
```
[GET|POST|PUT|DELETE] [المسار] مع [body أو query].
Success: [نص الاستجابة أو الحالة].
Failure: [كود الخطأ] مع [نص الخطأ].
```

---

### 2) عند إضافة صفحة أو سلوك واجهة في Frontend

| الخطوة | الإجراء |
|--------|---------|
| 1 | افتح `.tdad/workflows/frontend/frontend.workflow.json` — إن لم يكن للمجال مجلد، أضف عقدة `nodeType: "folder"` مع `folderPath: "frontend/اسم-المجال"`. |
| 2 | افتح أو أنشئ `.tdad/workflows/frontend/اسم-المجال/اسم-المجال.workflow.json`. |
| 3 | أضف عقدة `nodeType: "feature"`: |
| | • `id`: فعل-اسم (مثل `display-login-form`, `submit-login-credentials`) |
| | • `description`: [متى / عند ماذا] [ماذا يحدث]. Success: [ما يظهر]. Failure: [الرسالة أو السلوك]. |
| | • `testLayers`: `["ui"]` أو `["ui", "api"]` إن كان يستدعي API |
| 4 | إن كانت الصفحة تعتمد على تحميل بيانات، ضع العقدة التي تعرض البيانات في `dependencies` للعقدة التي تعتمد عليها (مثل زر الإرسال يعتمد على عرض النموذج). أضف `edges` داخل نفس الملف عند الحاجة. |

**قالب وصف UI:**
```
عند [حدث/نقرة/تحميل الصفحة]، [ماذا يحدث].
Success: [ما يظهر للمستخدم].
Failure: [الرسالة أو السلوك عند الخطأ].
```

---

### 3) عند تعديل سلوك موجود

| الخطوة | الإجراء |
|--------|---------|
| 1 | ابحث عن العقدة المقابلة في المجلد المناسب (backend أو frontend ثم المجال). |
| 2 | حدّث حقل `description` ليعكس السلوك الجديد (المحفّز، النتيجة، حالة الفشل). |
| 3 | إن ظهرت تبعية جديدة على عقدة أخرى، أضفها في `dependencies` وأضف `edge` إن كانت في نفس المجلد. |
| 4 | إن انقسم السلوك إلى سلوكين منفصلين، انسخ العقدة إلى عقدتين مع تحديث الـ id والوصف، وأضف تبعيات وحواف كما يلزم. |

---

### 4) عند إضافة مجلد (مجال) جديد بالكامل

| الخطوة | الإجراء |
|--------|---------|
| 1 | في `.tdad/workflows/backend/backend.workflow.json` أو `frontend/frontend.workflow.json` أضف عقدة جديدة: |
| | • `id`: اسم المجلد بـ kebab-case |
| | • `workflowId`: "backend" أو "frontend" |
| | • `title`: الاسم المعروض |
| | • `description`: وصف مختصر للمجال |
| | • `nodeType`: "folder" |
| | • `folderPath`: "backend/اسم-المجلد" أو "frontend/اسم-المجلد" |
| | • `position`: { x, y } |
| | • `dependencies`: [] |
| 2 | أنشئ المجلد والملف: `.tdad/workflows/backend/اسم-المجلد/اسم-المجلد.workflow.json` (أو frontend). |
| 3 | ضع في الملف الجديد `version`, `nodes` (عقد الميزات), `edges` (إن وجدت). ابدأ بعقدة واحدة على الأقل لكل سلوك رئيسي. |

---

## ما لا يُضاف كعقدة

- تثبيت حزم (`npm install`, `pnpm add`).
- إعداد متغيرات البيئة (`.env`).
- إعداد قاعدة البيانات أو الـ migrations وحدها.
- تكوين البناء (webpack, angular.json) دون سلوك قابل للاختبار.
- خطوات يدوية (مثل الحصول على API key من لوحة تحكم خارجية).

**العقد = سلوك يمكن كتابة سيناريو BDD أو اختبار Playwright/API له.**

## التحقق السريع بعد التعديل

1. **شغّل سكريبت التحقق** (من جذر المشروع):
   ```bash
   node .tdad/scripts/validate-workflows.mjs
   ```
   السكريبت يتحقق من:
   - **أخطاء البنية**: صحة JSON، الحقول الإلزامية، تكرار الـ id، الحواف تشير إلى عقد موجودة.
   - **انحرافات الواجهة**: كل مسار في `app.routes.ts` يجب أن يكون له مجلد workflow مطابق (مع خريطة المسارات).
   - **انحرافات الخادم**: كل عقدة مجلد في `backend.workflow.json` يجب أن يكون لها مجلد workflow فعلي؛ المجالات الرئيسية للـ API يجب أن تكون مغطاة بمجلد workflow.

2. **JSON صالح**: تأكد أن الملف يحوي JSON صحيحاً (لا تعليقات، لا فاصلة زائدة بعد آخر عنصر).
3. **الـ id فريد** داخل نفس الملف.
4. **الـ dependencies** تشير إلى `id` موجود في نفس الملف أو في مجلد آخر (لا حاجة لحافة cross-folder).
5. **الـ edges** فقط بين عقد في **نفس الملف**: `source` و `target` موجودان في `nodes` نفسها.

الخروج: `0` = نجاح، `1` = أخطاء أو انحرافات (يُستحسن إصلاحها قبل الـ commit).

## أين تكتب التحديثات

| نوع التغيير | الملف (أو الملفات) المحدَّثة |
|-------------|------------------------------|
| endpoint جديد في API | `backend/backend.workflow.json` (إن لزم مجلد جديد) + `backend/المجال/المجال.workflow.json` |
| صفحة أو مكون واجهة جديد | `frontend/frontend.workflow.json` (إن لزم مجلد جديد) + `frontend/المجال/المجال.workflow.json` |
| تعديل وصف أو تبعية ميزة موجودة | الملف الذي يحتوي عقدة تلك الميزة |
| إضافة مجال (backend أو frontend) | الملف الرئيسي (backend أو frontend) + إنشاء مجلد وملف المجال |

---

## خلاصة

- **أولاً بأول**: بعد كل ميزة أو endpoint قابل للاختبار، حدّث العقدة (أو أضف عقدة) في الملف المناسب.
- **وصف BDD**: كل عقدة لها وصف يوضح: المحفّز، النتيجة عند النجاح، والنتيجة عند الفشل.
- **لا عقد للإعداد فقط**: العقد للسلوك القابل للاختبار فقط.
- الالتزام بهذه الآلية يبقي هيكل TDAD والبروتوكول محدثين مع التطوير.
