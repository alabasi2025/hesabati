## 2. أخطاء تم اكتشافها وإصلاحها

### 2.1 انحراف مسارات API للرواتب (تم الإصلاح)

- **المشكلة:** الواجهة الأمامية كانت تستدعي مسارات **`/businesses/:bizId/salary-records`** و **`/salary-records/:id`** بينما الـ Backend يعرّف **`/businesses/:bizId/salaries`** و **`/salaries/:id`** فقط.
- **التأثير:** صفحة الرواتب كانت ستفشل بطلب 404 عند جلب/إضافة/تعديل/حذف سجلات الرواتب.
- **الإصلاح:** تم تعديل `frontend/src/app/services/api.service.ts` لاستخدام المسارات الصحيحة (`/salaries` بدلاً من `/salary-records`).

### 2.2 اختبار E2E للتبويب الجانبي (تم التحسين)

- **المشكلة:** الاختبار الأول كان يفشل لأن:
  - المحدد `.business-card` كان يطابق أيضاً بطاقة "إضافة عمل جديد" (التي قد تكون أول عنصر في بعض الحالات).
  - المحدد للـ sidebar كان عاماً جداً ولم يكن يطابق العنصر الفعلي `aside.sidebar`.
- **الإصلاح:**
  - استخدام `.business-card:not(.add-card)` لاستبعاد بطاقة الإضافة.
  - استخدام `aside.sidebar, .sidebar` للتبويب الجانبي وزيادة مهلة الظهور إلى 15 ثانية في الاختبار الأول.

## 3. فجوات وانحرافات محتملة (لم تُغيّر)

### 3.1 واجهة `getCollectionStyleConfig` / `saveCollectionStyleConfig`

- الواجهة الأمامية تستدعي `/businesses/:bizId/screens/:screenId/collection-style-config` (GET و PUT).
- الـ Backend يعرّف نفس المسارات — **لا انحراف**.

### 3.2 مسارات أخرى تم التحقق منها

- **تصنيفات المصروفات / ميزانية المصروفات / الرواتب:** مسارات الـ Backend تطابق استدعاءات الـ Frontend بعد تصحيح مسارات الرواتب.
- **العملات:** `GET /currencies` موجود في الـ Backend ومستدعى من الواجهة بشكل صحيح.

### 3.3 اختبارات الوحدة (Unit Tests)

- لا توجد ملفات `*.spec.ts` في المشروع سوى `e2e/sidebar.spec.ts` (Playwright).
- **فجوة:** عدم وجود unit tests لخدمات الواجهة أو للـ API في الـ Backend.

### 3.4 توثيق API

- **تم إضافة مواصفة OpenAPI 3.0** في `docs/openapi.yaml` تصف المسارات الرئيسية (المصادقة، الأعمال، المحطات، الحسابات، الصناديق، السندات، لوحة التحكم، الصحة). راجع `docs/API_DOCUMENTATION.md` لطريقة الاستخدام والتوسع.
- **فجوة متبقية:** يمكن توسيع المواصفة ليشمل بقية المسارات واستخدامها للتحقق التلقائي من تطابق الواجهة.

## 4. التوصيات

1. **تشغيل E2E بعد التصحيح:** تشغيل `npx playwright test e2e/sidebar.spec.ts` مرة أخرى للتأكد من نجاح كلا الاختبارين بعد تحسين المحددات والتوقيت.
2. **إضافة Unit Tests:** على الأقل لـ `ApiService` (مثلاً لمسارات الرواتب والمصروفات) وللوحدات الحرجة في الـ Backend.
3. **توثيق API:** تم إنشاء مواصفة OpenAPI في `docs/openapi.yaml`؛ يمكن توسيعها واستخدامها للتحقق التلقائي من تطابق الواجهة.
4. **التحقق الدوري:** تشغيل `validate-workflows.mjs` و `db:check` وبناء Frontend/Backend ضمن CI.
