# انحرافات الواجهة الأمامية (Frontend) والخادم الخلفي (Backend)

تاريخ الفحص: 2026-03-07

---

## 1. ملخص التطابق

- **العناوين (URLs):** مسارات الـ API في الواجهة تتطابق مع مسارات الـ Backend (مثل `/api/businesses`, `/api/businesses/:bizId/stations`, `/api/auth/login`, `/dashboard/stats`).
- **المصادقة:** الواجهة ترسل `Authorization: Bearer <token>` والـ Backend يتحقق منه عبر `authMiddleware()`.
- **شكل الاستجابة (المصادقة):** الـ Backend يعيد `{ token, user: { id, username, fullName, role } }` والواجهة تتوقعه نفس الشكل.
- **تطبيع المدخلات:** الـ Backend يستخدم `normalizeBody()` لقبول camelCase و snake_case، مما يقلل خطر عدم تطابق أسماء الحقول المرسلة من الواجهة.

## 2. انحرافات محتملة أو نقاط ضعف

### 2.1 تحليل جسم الاستجابة الناجحة في ApiService — **تم معالجته**

- **الموقع:** `frontend/src/app/services/api.service.ts` — الدالة `request<T>()`.
- **المشكلة السابقة:** في حالة النجاح كان يتم استدعاء `return res.json()` فقط؛ فإذا أعاد الـ Backend استجابة بدون جسم كان يحدث خطأ `Unexpected end of JSON input`.
- **الإصلاح:** تم الاعتماد على `res.text()` أولاً، ثم تحليل JSON فقط عند وجود نص، مع إرجاع `undefined` عند جسم فارغ وتجنب رمي استثناء. نفس النهج يُستخدم لتحليل جسم الخطأ لتجنب أخطاء JSON عند استجابة خطأ غير صالحة.

### 2.2 مسارات التعديل والحذف بدون bizId (Legacy)

- **الواجهة:** تستخدم أحياناً مسارات بدون `bizId`، مثل:
  - `updateEmployee(id, d)` → `PUT /employees/:id`
  - `updateAccount(id, d)` → `PUT /accounts/:id`
  - `deleteAccount(id)` → `DELETE /accounts/:id`
  - `updatePartner(id, d)` → `PUT /partners/:id`
  - وغيرها.
- **الـ Backend:** يوفر هذه المسارات كـ "legacy" ويستنتج العمل من المورد (مثلاً من الحساب أو الموظف).
- **الانحراف:** لا يوجد خطأ وظيفي، لكن هناك ازدواجية بين:
  - مسارات مرتبطة بعمل معين: `PUT /businesses/:bizId/accounts/:id`
  - مسارات بدون bizId: `PUT /accounts/:id`
- **التوصية:** توحيد الاستخدام على المدى البعيد (مثلاً الاعتماد على مسارات تحت `businesses/:bizId` فقط) لتقليل الالتباس وصيانة الأمان والصلاحيات.

### 2.3 إنشاء حساب فوترة للموظف (بدون bizId في الرابط)

- **الواجهة:** `createEmployeeBillingAccount(d)` → `POST /employee-billing-accounts` مع إرسال الجسم فقط (بدون `bizId` في المسار).
- **الـ Backend:** يستقبل الطلب ويستنتج العمل من `employeeId` الموجود في الجسم.
- **الوضع:** متطابق؛ الـ Backend يتحقق من الصلاحية عبر الموظف. يُذكر هنا للتوثيق وليس كخطأ.

### 2.4 شكل رسائل الخطأ

- **الـ Backend:** يعيد عادةً `{ error: string }` وأحياناً `details` أو `location`.
- **الواجهة:** في `api.service.ts` تتعامل مع `err.error` و`err.details` و`err.location` وتبني رسالة عربية.
- **الانحراف:** إذا استخدم أي endpoint في الـ Backend شكلاً مختلفاً لرسالة الخطأ (مثلاً `message` بدل `error`)، قد لا تظهر الرسالة الصحيحة في الواجهة.
- **التوصية:** توحيد شكل استجابة الخطأ في الـ Backend (مثلاً دائماً `error` مع اختياري `details`/`location`) وتوثيقه.

### 2.5 لوحة التحكم (Dashboard) — نوع totalSalaries

- **الواجهة:** `DashboardStats` يعرّف `totalSalaries: string`.
- **الـ Backend:** يعيد `totalSalaries: salaryResult?.[0]?.total || '0'` (نوع string من الاستعلام).
- **الوضع:** متطابق.

## 3. نقاط إيجابية

- استخدام **نفس الأساس** للـ API في الواجهة (`/api`) مع الـ proxy إلى المنفذ 3000.
- الـ Backend يطبّق **تطبيع الجسم** (`normalizeBody`) ليدعم camelCase و snake_case ويقلل انحراف أسماء الحقول.
- وجود مسارات **legacy** في الـ Backend يسمح للواجهة باستخدام إما المسارات المرتبطة بـ `bizId` أو المسارات بالمعرّف فقط دون كسر الوظيفة الحالية.
- تطابق أنواع الأعمال (`stations` | `single_station` | `personal`) وحقل `type` في استجابة الأعمال مع واجهة الواجهة.
