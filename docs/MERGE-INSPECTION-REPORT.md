## 3. الفرق الجوهري بين المحلي والبعيد (تعارض متوقع)

### 3.1 api.rest.ts — تعارض كبير

- **المحلي (عندك):** تم **تقسيم** المسارات إلى أكثر من **12 ملفاً** منفصلاً:  
  `reporting.routes.ts`, `workflow.routes.ts`, `inventory.routes.ts`, `ui-builder.routes.ts`,  
  `warehouse.routes.ts`, `accounts.routes.ts`, `partners.routes.ts`, `pending-settlements.routes.ts`,  
  `categories-expenses.routes.ts`, `operation-types.routes.ts`, `journal-entries.routes.ts`,  
  `billing-config.routes.ts`، وتبقى في `api.rest.ts` عدد محدود من المسارات.
- **البعيد:** يملك نسخة **موحّدة** من `api.rest.ts` (~4439 سطر) **بدون** هذا التقسيم.

**النتيجة:** إذا نفذت `git pull` (أو merge) كما هو:
- Git سيعتبر أن `api.rest.ts` متغيّر في الطرفين؛ سيحدث **تعارض دمج (merge conflict)** في هذا الملف.
- إذا قبلت نسخة البعيد، ستفقد **كل التقسيم** الذي تم محلياً وتعود لملف ضخم واحد.
- إذا قبلت نسخة المحلي، ستفقد **أي تعديلات** داخل api.rest من البعيد (مثل تحسينات مرتبطة بالسندات/القيود إن وُجدت في نفس الملف).

### 3.2 funds.routes.ts

- **البعيد:** يضيف منطق ترقيم محسّن (يدوي/تلقائي) و`ensureCounterAtLeast` و`fundTypes`.
- **المحلي:** قد يكون نسخة أقدم بدون هذه الإضافات.

هنا الدمج عادة أسهل: يمكن دمج تغييرات البعيد في `funds.routes.ts` مع الاحتفاظ ببنية الملف الحالية.

### 3.3 sequencing.ts و transaction.service.ts

- تغييرات البعيد (تصدير `TYPE_PREFIXES`, `generateItemCode`، وواجهات المعاملات متعددة السطور) **مفيدة ولا تتعارض مع فكرة التقسيم**؛ يفضّل أخذها مع الدمج.

## 4. التوصية قبل تنفيذ الدمج

### 4.1 فحص دقيق (تم)

- تم جلب التحديثات (`git fetch origin`).
- تم عرض الـ commits والملفات المتغيرة (`git log main..origin/main`, `git diff main..origin/main`).
- تم توثيق التعارض الرئيسي (api.rest.ts) والملفات الأخرى.

### 4.2 خيارات آمنة للدمج

1. **الخيار (أ) — دمج مع الاحتفاظ بالتقسيم (موصى به)**  
   - تنفيذ: `git pull origin main` (أو merge من `origin/main`).  
   - عند ظهور تعارض في **api.rest.ts**:  
     - اختر **نسخة المحلي** (التي تحتوي التقسيم) وحل التعارض يدوياً بحيث يبقى هيكل المسارات المقسم كما هو.  
     - ثم راجع أي تحسينات من البعيد كانت داخل api.rest (مثلاً نصوص أو شروط) وانقلها يدوياً إلى الملف المناسب من ملفات الـ routes إذا لزم.  
   - في **funds.routes.ts**: اقبل تغييرات البعيد (أو دمجها يدوياً) للحصول على الترقيم المحسّن ورقم الخزينة اليدوي.  
   - في **sequencing.ts** و **transaction.service.ts**: اقبل تغييرات البعيد (أو دمجها) للاستفادة من `TYPE_PREFIXES`, `generateItemCode` والواجهات الجديدة.

2. **الخيار (ب) — دمج ثم إعادة تطبيق التقسيم**  
   - دمج البعيد أولاً وقبول نسخة البعيد لـ api.rest مؤقتاً.  
   - ثم إعادة استخراج المسارات من api.rest إلى الملفات المقسمة من جديد (أو استعادة الملفات من commit محلي سابق قبل الدمج ثم دمج التحسينات من البعيد فيها).  
   - هذا يتطلب جهداً أكبر ويُستخدم إن رغبت في الحصول على نسخة البعيد كأساس ثم إعادة الهيكلة.

3. **الخيار (ج) — عدم الدمج الآن**  
   - الإبقاء على الفرع الحالي ومواصلة العمل على التقسيم أو الميزات.  
   - لاحقاً يمكن تنفيذ دمج مع حل تعارض api.rest كما في (أ).

### 4.3 خطوة تنفيذية مقترحة (لخيار أ)

```bash
# 1) التأكد من وجود commit محلي للتقسيم (إن لم يكن مرفوعاً بعد)
git status
git add backend/src/routes/api/*.routes.ts backend/src/routes/api/index.ts docs/
git commit -m "refactor: split api.rest into multiple route modules"

# 2) سحب التحديثات من البعيد
git pull origin main

# 3) عند التعارض في api.rest.ts:
#    - فتح الملف وحل التعارض (Conflict markers)
#    - الاحتفاظ بهيكل المحلي (المسارات المقسمة) ونسخ أي تحسينات مفيدة من البعيد إلى الملفات المناسبة
# 4) في funds.routes.ts: مراجعة الدمج وقبول منطق الترقيم من البعيد إن لم يكن مدمجاً تلقائياً بشكل صحيح
# 5) تشغيل الاختبارات
cd backend && pnpm test
```

## 5. الخلاصة

| البند | الحالة |
|-------|--------|
| جلب التحديثات من البعيد | تم (`git fetch`) |
| فحص الـ commits والملفات | تم وموثّق أعلاه |
| تعارض رئيسي | **api.rest.ts**: نسخة ضخمة موحّدة (بعيد) مقابل نسخة مقسمة (محلي) |
| توصية | دمج مع **الاحتفاظ بنسخة المحلي لـ api.rest** وحل التعارض يدوياً، وأخذ تحسينات **funds.routes.ts** و **sequencing.ts** و **transaction.service.ts** والواجهات الأمامية من البعيد |
| تنفيذ الدمج الفعلي | يفضّل بعد عمل نسخة احتياطية أو commit محلي، ثم تنفيذ الخطوات في 4.3 |

تم إعداد هذا التقرير بعد فحص دقيق للتحديثات القادمة من المستودع البعيد دون تنفيذ الدمج بعد، لاتخاذ قرار مدروس قبل الدمج.
