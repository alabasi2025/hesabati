## 1. نظرة معمارية عالية المستوى

### 1.1 نمط المعمارية

- **نمط:** تطبيق ويب متعدد المستخدمين (Multi-tenant SPA + API).
- **الحدود:**
  - **عميل ويب (Frontend):** تطبيق Angular SPA يتصل بـ API عبر HTTP و WebSocket.
  - **خادم API (Backend):** تطبيق Node (Hono) يوفر REST و WebSocket، ووحدة عمل واحدة (Business) لكل طلب بعد المصادقة.
  - **قاعدة البيانات:** PostgreSQL؛ الوصول عبر Drizzle ORM داخل الـ Backend فقط.

### 1.2 التقسيم المنطقي (High-Level)

```
┌─────────────────────────────────────────────────────────────────┐
│                        المستخدمون (متصفح)                         │
└───────────────────────────────┬─────────────────────────────────┘
                                │ HTTPS / WSS
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│  Frontend (Angular)  │  Proxy → /api, /health → Backend          │
└───────────────────────────────┬─────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│  Backend (Hono + Node)                                          │
│  ├── Auth / bizAuth                                              │
│  ├── API Routes (REST) + WebSocket                               │
│  ├── Engines (Transaction, Workflow, Inventory, Reporting, …)    │
│  └── DB (Drizzle → PostgreSQL)                                  │
└───────────────────────────────┬─────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│  PostgreSQL                                                      │
└─────────────────────────────────────────────────────────────────┘
```

## 2. المكدس التقني (Tech Stack)

| الطبقة | التقنية | الملاحظات |
|--------|---------|------------|
| **Frontend** | Angular 21, TypeScript, Tailwind, Chart.js | SPA مع توجيه وخدمات وواجهات موحدة. |
| **Backend** | Node.js, Hono, TypeScript (ESM), tsx | خفيف، بدون إطار ثقيل؛ مناسب لـ API و WebSocket. |
| **قاعدة البيانات** | PostgreSQL | بيانات علائقية؛ الهجرات والـ schema عبر Drizzle. |
| **ORM / Schema** | Drizzle ORM | Schema في الكود، db:push أو migrate. |
| **مصادقة** | JWT (Bearer) | مع التحقق من العمل (bizId) بعد تسجيل الدخول. |
| **اتصال فوري** | WebSocket (مدمج في نفس الخادم) | للإشعارات والتحديثات الحية إن وُجدت. |

## 3. مبادئ المعمارية والقرارات

### 3.1 محرك واحد للمعاملات المالية

- **القرار:** كل تأثير محاسبي (قيد، أرصدة، صناديق) يمر عبر **المحرك المالي** (`transaction.service.ts`): `postTransaction`, `confirmDraftVoucher`, `applyAccountingForConfirmedVoucher`.
- **المنطق:** تجنب ازدواجية القيد والأرصدة، ضمان ترقيم وتدقيق وملكية موحّدة.
- **التطبيق:** اعتماد المسودات (من التحسينات أو من سير العمل) يستدعي دوال المحرك ولا ينفذ SQL القيد يدوياً.

### 3.2 عزل العمل (Tenant = Business)

- **القرار:** كل طلب API (بعد المصادقة) مرتبط بعمل واحد (`bizId`). الحسابات والصناديق والسندات والعمليات المخزنية تخص هذا العمل فقط.
- **الآلية:** `bizAuthMiddleware()` وفحص ملكية الموارد (مثل `verifyAccountOwnership`) قبل أي معاملة.

### 3.3 أنواع العمليات كقوالب (Operation Types)

- **القرار:** جدول `operation_types` يحدد القوالب (حساب مصدر، صندوق، نوع سند، سير عمل، قيد آلي للعمليات المخزنية). الترقيم والصلاحيات تعتمد على `operationTypeId`.
- **التكامل:** المحرك المالي، سير العمل، العمليات المخزنية، والصلاحيات تستخدم نفس القالب؛ تحقق الصلاحيات (`validateConstraints`) يُستدعى عند إنشاء السندات والعمليات المخزنية.

### 3.4 طبقة الخدمات (Engines) وعدم تكرار المنطق

- **القرار:** المنطق الأساسي في **خدمات** (محرك مالي، مخزني، ترقيم، عملات، تقارير، واجهات ديناميكية، صلاحيات). مسارات الـ API تستدعي الخدمات ولا تكرر قواعد العمل.
- **الاستثناءات المُدارة:** استعلامات مخصصة في محرك بناء الواجهات مقيدة بـ SELECT وحماية نصية؛ يُوصى بتقييد إنشاءها بصلاحيات إدارية.

## 4. الحدود والتبعيات

| المكون | يعتمد على | لا يعتمد على |
|--------|-----------|---------------|
| **Frontend** | API (REST + WebSocket)، شكل الاستجابة والمصادقة | تفاصيل قاعدة البيانات أو لغة الـ Backend |
| **Backend API** | الخدمات (Engines)، Drizzle، PostgreSQL | تفاصيل واجهة المستخدم أو تقنيات الواجهة |
| **المحرك المالي** | Drizzle (جداول القيود والأرصدة)، الترقيم، التحقق من الملكية | مسارات HTTP أو تفاصيل الواجهة |
| **محرك المخزون** | جداول المخزون والحركات | المحرك المالي (التكامل عبر مسار API عند تفعيل القيد الآلي) |

## 5. المخاطر والتخفيف (من منظور المعمارية)

| المخاطر | التخفيف |
|---------|----------|
| **تسريب بيانات بين الأعمال** | فحص ملكية الموارد في المحرك المالي والـ API؛ ربط كل طلب بـ bizId. |
| **ازدواجية منطق القيد** | توحيد الاعتماد عبر المحرك المالي فقط (تم تطبيقه). |
| **استعلامات مخصصة غير موثوقة** | حصر الاستعلامات بـ SELECT وفحوصات نصية؛ تقييد إنشاء/تعديل المصادر بصلاحيات إدارية. |
| **JWT وبيانات الإنتاج** | عدم hardcode JWT_SECRET أو كلمة مرور DB في الكود؛ استخدام متغيرات بيئة في الإنتاج. |

## 6. التوسع والصيانة

- **إضافة محرك جديد:** إنشاء خدمة في `backend/src/services/` وربطها من مسارات الـ API دون تكرار منطق موجود في محرك آخر.
- **إضافة نوع عملية:** استخدام جدول `operation_types` وربط الصلاحيات والترقيم وسير العمل بالمعرّف.
- **توحيد مسارات API:** التوصية بالاعتماد تدريجياً على مسارات تحت `businesses/:bizId` وتقليل المسارات الـ legacy بدون bizId (انظر FRONTEND-BACKEND-DEVIATIONS.md).
