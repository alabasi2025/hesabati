# صفحة أنواع العمليات (Operation Types)

## الغرض

صفحة أنواع العمليات هي المكان المركزي لإنشاء وإدارة **القوالب (Templates)** و**التصنيفات (Categories)**. القوالب هي عمليات جاهزة تسهّل على المستخدم تنفيذ العمليات بسرعة ودقة. تحتوي الصفحة على تبويبين مستقلين:

1.  **القوالب**: لإدارة القوالب الجاهزة التي تُستخدم في إنشاء السندات.
2.  **التصنيفات**: لإدارة التصنيفات التي تُستخدم لتنظيم القوالب، وتُدار بشكل مستقل عن القوالب.

> **ملاحظة مستقبلية:** حالياً النظام يدعم القالب المالي فقط. لاحقاً سيتم التوسع لدعم أنواع أخرى (مخزني، مخزني مالي، وغيرها).

---

## إنشاء قالب جديد (Wizard - 6 خطوات)

يتم إنشاء قالب جديد عبر معالج (Wizard) من **6 خطوات**:

| الخطوة | العنوان | الوصف |
|--------|---------|-------|
| **1** | **اختيار التصنيف** | يختار المستخدم تصنيفاً للقالب من قائمة التصنيفات الموجودة مسبقاً. |
| **2** | **نوع السند** | يحدد نوع السند: **قبض**، **صرف**، أو **قيد محاسبي**. |
| **3** | **وسيلة الدفع** | تظهر فقط في حال اختيار **قبض** أو **صرف**. يختار المستخدم بين: **نقداً**، **بنك**، **صراف**، أو **محفظة**. |
| **4** | **المصدر (الطرف الأول)** | يعتمد على وسيلة الدفع المختارة في الخطوة 3. يختار المستخدم الحساب المصدر المحدد (أي صندوق؟ أي بنك؟ إلخ). |
| **5** | **الحسابات المرتبطة (الطرف الثاني)** | عرض ديناميكي لجميع الحسابات من صفحة الحسابات، مع فلاتر حسب النوع. |
| **6** | **الاسم والتفاصيل** | يحدد المستخدم اسم القالب، أيقونة، لون، ووصف اختياري. |

> **ملاحظة:** عند اختيار **قيد محاسبي** في الخطوة 2، يتم تخطي الخطوتين 3 (وسيلة الدفع) و 4 (المصدر) مباشرةً.

---

## تفاصيل الخطوات

### الخطوة 3: وسيلة الدفع

تظهر هذه الخطوة فقط عند اختيار **قبض** أو **صرف** في الخطوة 2. يختار المستخدم إحدى وسائل الدفع التالية:

| وسيلة الدفع | القيمة |
|-------------|--------|
| نقداً | cash |
| بنك | bank |
| صراف | exchange |
| محفظة إلكترونية | e_wallet |

### الخطوة 4: المصدر (الطرف الأول)

وسيلة الدفع التي يتم اختيارها في الخطوة 3 تحدد بشكل مباشر نوع الحسابات التي ستظهر في هذه الخطوة:

| وسيلة الدفع | الحسابات التي تظهر كمصدر | الحقل المُخزَّن |
|-------------|--------------------------|-----------------|
| **نقداً** (cash) | الصناديق (Funds) | `source_fund_id` |
| **بنك** (bank) | حسابات البنوك | `source_account_id` |
| **صراف** (exchange) | حسابات الصرافين | `source_account_id` |
| **محفظة إلكترونية** (e_wallet) | حسابات المحافظ | `source_account_id` |

### الخطوة 5: الحسابات المرتبطة (الطرف الثاني)

في هذه الخطوة، يتم جلب قائمة الحسابات من الـ API Endpoint `getAllAccounts` مع استخدام المعامل `?all=true`. هذا الـ Endpoint يرجع قائمة مدمجة تحتوي على كل من:

- الحسابات البنكية والصرافين والمحافظ (`accounts`).
- الصناديق (`funds`).
- حسابات الفوترة (`billing`).

تُعرض هذه القائمة للمستخدم مع فلاتر حسب النوع، ليختار منها الحسابات التي يريد ربطها بالقالب كطرف ثانٍ. هذه القائمة هي نفسها التي تظهر في صفحة الحسابات.

## البنية في قاعدة البيانات

### جدول `operation_types`

| الحقل | النوع | الوصف |
|-------|-------|-------|
| id | serial | المعرف |
| business_id | integer | ربط بالعمل |
| name | varchar(200) | اسم القالب |
| description | text | وصف اختياري |
| icon | varchar(50) | أيقونة |
| color | varchar(20) | لون |
| category | varchar(50) | التصنيف الذي ينتمي إليه القالب (ينشئه المستخدم يدوياً) |
| voucher_type | varchar(30) | نوع السند (payment/receipt/journal) |
| payment_method | varchar(30) | وسيلة الدفع (cash/bank/exchange/e_wallet) - للقبض والصرف فقط |
| **source_account_id** | **integer** | **الطرف الأول للبنك/صراف/محفظة. FK → accounts.id** |
| **source_fund_id** | **integer** | **الطرف الأول للصندوق. FK → funds.id** |
| screens | text | الشاشات المخصصة التي يظهر فيها |
| requires_attachment | boolean | يتطلب مرفق |
| has_multi_lines | boolean | يدعم أسطر متعددة |
| sort_order | integer | ترتيب العرض |
| is_active | boolean | نشط |

### جدول `operation_type_accounts`

هذا الجدول يمثل **الحسابات المرتبطة (الطرف الثاني)** التي يتم اختيارها في الخطوة 5.

| الحقل | النوع | الوصف |
|-------|-------|-------|
| id | serial | المعرف |
| operation_type_id | integer | ربط بالقالب |
| account_id | integer | حساب عادي |
| employee_billing_account_id | integer | حساب فوترة موظف |
| label | varchar(200) | تسمية الحساب |
| permission | varchar(30) | الصلاحية (both/receive_only/pay_only) |
| sort_order | integer | الترتيب |

## الـ API Endpoints

### القوالب

| Endpoint | الوظيفة |
|----------|---------|
| GET /businesses/:bizId/operation-types | جلب كل القوالب (مع فلتر حسب category أو screen) |
| GET /operation-types/:id | جلب قالب واحد بالتفصيل |
| POST /businesses/:bizId/operation-types | إنشاء قالب جديد مع حساباته |
| PUT /businesses/:bizId/operation-types/:id | تعديل قالب |
| DELETE /businesses/:bizId/operation-types/:id | حذف قالب وحساباته |

### التصنيفات

| Endpoint | الوظيفة |
|----------|---------|
| GET /businesses/:bizId/operation-categories | جلب كل التصنيفات |
| POST /businesses/:bizId/operation-categories | إنشاء تصنيف جديد |
| PUT /businesses/:bizId/operation-categories/:id | تعديل تصنيف |
| DELETE /businesses/:bizId/operation-categories/:id | حذف تصنيف |

### الحسابات (للخطوة 5)

| Endpoint | الوظيفة |
|----------|---------|
| GET /businesses/:bizId/accounts?all=true | جلب كل الحسابات المدمجة (accounts + funds + billing) |

---

## ملاحظات

- حقل `category` يجب أن يكون ديناميكياً (ينشئه المستخدم من تبويب التصنيفات) وليس قائمة ثابتة في الكود.
- تم إضافة حقلي `source_account_id` و `source_fund_id` لتخزين الطرف الأول بشكل صريح في قاعدة البيانات بدلاً من الاستنتاج منطقياً من وسيلة الدفع.
- الخطوتان 3 و 4 يتم تخطيهما تلقائياً عند اختيار **قيد محاسبي** كنوع للسند.
- الخطوة 5 تعتمد على `getAllAccounts API` مع `?all=true` لعرض قائمة موحدة ومدمجة تطابق ما يظهر في صفحة الحسابات.
- القوالب تُستخدم لاحقاً داخل الشاشات المخصصة لتنفيذ العمليات.
