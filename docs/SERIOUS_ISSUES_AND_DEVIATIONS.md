## 2. انحرافات ومشاكل موثقة (من تقارير سابقة)

### 2.1 انحراف التبويب الجانبي: «ملخص الأعمال» — **تم إصلاحه**

- **الوضع المطلوب:** عنصر «ملخص الأعمال» يتبع **أعمال شخصية** (حسابات المالك الشخصية وأعماله الخاصة) فقط.
- **ما تم سابقاً:** في الواجهة (القائمة الافتراضية) تم إظهار «ملخص الأعمال» فقط في عمل **أعمال شخصية** وإزالته من **المحطات** و**محطة واحدة**.
- **الإصلاح المنفّذ (2026-03-05):**
  1. **Seed:** في `seed.ts`, `seed_sidebar.ts`, `reset-sidebar.ts` تمت إزالة «ملخص الأعمال» من قسم الرئيسية لأعمال **المحطات** و**محطة واحدة**، وإضافته فقط في الفرع `else` (أعمال شخصية).
  2. **API:** في مسار `GET /businesses/:bizId/users/:userId/sidebar` تمت إضافة فلتر: استبعاد العنصر الذي `screenKey === 'summary'` عندما يكون نوع العمل (من جدول `businesses.type`) ليس `personal`. بهذا لا يظهر «ملخص الأعمال» للمحطات حتى لو كانت البيانات القديمة في DB تحتوي عليه.
- **النتيجة:** «ملخص الأعمال» يظهر فقط لأعمال شخصية سواء من القائمة الافتراضية أو من التبويب المحمّل من الـ API.

### 2.2 بروتوكول TDAD (توثيق)

- **التقرير:** `.tdad/PROTOCOL_AUDIT_REPORT.md` يذكر:
  - مسارات API غير موثقة في البروتوكول (مثل POST/PUT/DELETE للرواتب، PUT/DELETE للحسابات، عكس السند).
  - عقد عامة تحتاج تقسيم (مثلاً لوحة التحكم، إنشاء سند).
  - أوصاف ناقصة لبعض العقد وسلوكيات واجهة غير موثقة.
- **التأثير:** توثيقي فقط؛ لا يمنع تشغيل النظام أو الاختبارات الآلية.
- **الإصلاح الموصى به:** تنفيذ خطة الإصلاح في `PROTOCOL_AUDIT_REPORT.md` (إضافة العقد المفقودة، تقسيم العقد العامة، تحسين الأوصاف).

### 2.3 اختبارات وتوثيق API

- **Unit tests:** لا توجد اختبارات وحدة لخدمات الواجهة أو لجميع مسارات الـ API في الـ Backend (ما عدا ما هو مغطى في `api.integration.test.ts` وغيرها).
- **توثيق API:** لا يوجد ملف OpenAPI/Swagger يصف مسارات الـ API.
- **التأثير:** صعوبة أكبر في تتبع التوافق بين الواجهة والـ Backend وفي اكتشاف الأخطاء عند التعديل.
- **التوصية:** إضافة unit tests للوحدات الحرجة وتوليد/تحديث مواصفة OpenAPI من مسارات الـ Backend.
